<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser JS → access token storage | XSS-exposed surface; token must stay in memory (Zustand), never localStorage (D-05). |
| client role-gating (ProtectedRoute) → server | UI gating is cosmetic; the authoritative check is the backend 403. Treating UI gating as security is the anti-pattern. |
| browser → /auth/refresh (cookie) | httpOnly refresh cookie auto-sent cross-port; CORS/withCredentials must align. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-XSS-TOKEN | Information Disclosure | access-token storage | mitigate | Access token in Zustand memory only; no `persist`, no localStorage write. Acceptance grep + useSilentRefresh.test assert it. |
| T-01-UIGATE | Elevation of Privilege | ProtectedRoute as "security" | mitigate | ProtectedRoute is UX-only; real enforcement is Plan 02's `require_role` 403 (re-verified by the checkpoint's curl step). Reminder comment kept in source. |
| T-01-REFRESH-RACE | Denial of Service | concurrent 401s → multiple refreshes | mitigate | `isRefreshing` + `failedQueue` ensures a single `/auth/refresh` per burst (RESEARCH Pattern 7). |
| T-01-CSRF | Tampering | cookie-auth refresh endpoint | mitigate | SameSite=Lax cookie (Plan 02) + access token sent via Authorization header (not cookie) → bulk API is CSRF-resistant. |

No HIGH-severity threat is left unmitigated; the role-bypass enforcement (T-01-UIGATE) is verified server-side in the blocking checkpoint.
</threat_model>
