# Phase 1 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Greenfield note:** This is the project's first UI phase — no `frontend/` directory, no `components.json`, no design tokens exist yet. This contract establishes the FOUNDATIONAL design system that every later phase (Catalog, Borrowing, Loans, Notifications) inherits. The executor scaffolds the frontend (per 01-RESEARCH.md's recommended structure) and applies this contract during that scaffold — it is not optional polish, it is the template.
>
> All values below are **Claude's Discretion defaults** (CONTEXT.md explicitly defers visual/implementation specifics not otherwise locked). They are deliberately conservative, institutional, and consistent with an admin-heavy CRUD tool serving both students and librarians. Flag any of these to the user for override before/while executing if preferred.

## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn |
| Preset | style: `new-york`, base color: `slate`, CSS variables: yes (enables theming via CSS custom properties — required for any future dark-mode or per-role theming work) |
| Component library | radix (shadcn/ui is built on Radix UI primitives — accessible by default, critical for forms with validation states) |
| Icon library | `lucide-react` (shadcn's default pairing; consistent icon set across all later phases) |
| Font | System font stack — Tailwind v4 default `font-sans` (`ui-sans-serif, system-ui, -apple-system, "Segoe UI", Roboto, ...`). Zero load cost, no FOUT/FOIT, renders natively on every OS. Appropriate for an internal university tool where "fast and clean" beats "branded and designed." |

**Setup commands for executor (run during frontend scaffold, after `npm create vite@latest frontend -- --template react-ts`):**
```bash
npx shadcn@latest init
# When prompted: style = new-york, base color = slate, CSS variables = yes
npx shadcn@latest add button input label form card alert separator
```
These six primitives cover every component this phase needs (login/signup/forgot-password/reset-password forms + protected-route stub dashboard). Later phases add to this set incrementally — do not bulk-install the entire shadcn catalog up front.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, input internal padding fine-tuning |
| sm | 8px | Compact element spacing (label-to-input gap, button icon gap) |
| md | 16px | Default element spacing (form field vertical rhythm, card padding) |
| lg | 24px | Section padding (card body padding, form group spacing) |
| xl | 32px | Layout gaps (space between form card and page edge on desktop) |
| 2xl | 48px | Major section breaks (vertical centering offset for auth pages) |
| 3xl | 64px | Page-level spacing (top margin on standalone auth pages) |

Exceptions: Touch targets for icon-only buttons (e.g., password show/hide toggle inside an input) use a 44×44px minimum hit area even though the visible icon is smaller — accessibility requirement (WCAG 2.5.5), not a spacing-scale violation since it governs the interactive hit region, not visual spacing.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 (regular) | 1.5 |
| Label | 14px | 500 (medium) | 1.5 |
| Heading | 20px | 600 (semibold) | 1.2 |
| Display | 28px | 600 (semibold) | 1.2 |

**Usage map for this phase:**
- Display (28/600/1.2): Page titles — "Sign up", "Log in", "Reset your password", "Check your email"
- Heading (20/600/1.2): Card section headers (e.g., "Create your account"), dashboard stub headings
- Body (16/400/1.5): Form helper text, paragraph copy, error/success messages, dashboard stub body content
- Label (14/500/1.5): Form field labels, input hints, links ("Already have an account? Log in")

Exactly 2 weights declared (400 regular, 600 semibold) plus one supporting medium (500) reserved ONLY for form labels — do not introduce additional weights. This 3-weight allowance (with 500 scoped narrowly) keeps the type system simple while giving labels enough visual distinction from body copy in dense forms.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `slate-50` (`#f8fafc`) light / page background, form page canvas | Page backgrounds, the surface behind auth cards |
| Secondary (30%) | `white` (`#ffffff`) with `slate-200` (`#e2e8f0`) borders | Auth form cards, dashboard stub shell, nav/sidebar surfaces (later phases) |
| Accent (10%) | `blue-600` (`#2563eb`) | **Reserved for:** primary CTA buttons (Sign up / Log in / Send reset link / Reset password / Save), active focus rings on inputs, the active/current nav indicator in the dashboard shell, links to switch between login/signup ("Don't have an account? Sign up") |
| Destructive | `red-600` (`#dc2626`) | **Reserved for:** inline validation error text and error-state borders on inputs, the "Log out of all sessions" warning copy shown after password reset (per D-07), and any future destructive-action buttons (e.g., "Reject request" in Phase 4) — establishes the pattern now |

Accent reserved for: primary submit buttons on auth forms, input focus-ring color, the "switch form" links (login↔signup↔forgot-password), and the active-state indicator in the role-gated dashboard nav shell. Do NOT use accent color for body text, card backgrounds, borders on non-interactive elements, or decorative purposes — this keeps the 10% genuinely scarce and meaningful (a student or librarian should be able to scan a page and immediately spot the one actionable thing).

**Dark mode:** Out of scope for this phase (not requested in CONTEXT.md or REQUIREMENTS.md). The `CSS variables: yes` shadcn preset choice keeps the door open for a future phase to add a dark theme without restructuring — but do not build dark-mode toggle UI now.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Per-form, specific verb + noun: **"Sign up"** (signup form) · **"Log in"** (login form) · **"Send reset link"** (forgot-password form) · **"Reset password"** (reset-password form) · **"Log out"** (session menu — not destructive, no confirmation) |
| Empty state heading | Not applicable to this phase — auth has no list/data views. The role-gated dashboard stub shows a static welcome, not an empty state. First real empty state appears in Phase 2 (catalog search with no results). |
| Empty state body | N/A (see above) — dashboard stub copy instead: **"Welcome, {first segment of email}"** with role-specific subtext: Student → *"Search the catalog to find your next book."* / Librarian → *"Manage requests and the catalog from here."* (both are placeholders proving role-differentiated rendering for AUTH-04's success criterion; full functionality lands in later phases) |
| Error state | **Login failure:** "Invalid email or password. Check your details and try again." (deliberately generic — does not reveal whether the email exists, mirrors D-09's enumeration-safety principle for consistency across all auth surfaces) · **Signup — invalid librarian code (D-03 exact copy):** "Invalid librarian code — check with your library administrator, or sign up as a student." · **Signup — email already registered:** "An account with this email already exists. Log in instead, or use a different email." · **Forgot-password (always shown, per D-09):** "If that email is registered, you'll receive a reset link shortly." · **Reset link expired/used:** "This reset link is no longer valid. It may have expired or already been used — request a new one." · **Generic network/server error (any form):** "Something went wrong on our end. Please try again in a moment." |
| Destructive confirmation | **Password reset completion (D-07 — revokes ALL sessions):** Surfaced as an informational notice on the reset-password form itself (not a separate confirm dialog, since the user is already mid-recovery-flow and the action is the *point* of the flow): *"Resetting your password will sign you out of all devices, including this one once you're redirected — for your security."* No additional confirm/cancel step (would add friction to a security-critical recovery moment); the notice sets expectations before submission. **"Log out" (current session only, per D-06):** No confirmation needed — low-stakes, reversible (just log back in), other sessions stay active. This establishes the pattern: reserve confirmation dialogs for irreversible/high-blast-radius actions (e.g., Phase 3's "delete book", Phase 4's "reject request") — do not over-confirm low-stakes actions like a single-session logout. |

## Interaction & State Contract

*(Phase-specific addendum — beyond the template's standard sections, capturing the auth-specific behavioral contract the executor needs.)*

| Surface | States to implement |
|---------|---------------------|
| Login / Signup / Forgot-password / Reset-password forms | `idle` → `submitting` (disable submit button, show inline spinner inside the button — do not replace form with a full-page spinner) → `success` (redirect or success message per flow) → `error` (inline message above or below the form, using Destructive red text, role="alert" for screen readers) |
| Silent refresh on app load (D-05) | MUST be invisible to the user — no flash of login screen, no visible spinner on the root route. Render nothing (or a minimal blank canvas matching the page background `slate-50`) until the refresh check resolves, then route to the correct authenticated/unauthenticated view. A loading spinner here would itself be the "jarring flash" the success criteria warns against — prefer a brief blank canvas over a spinner flash. |
| Password field | Includes a show/hide toggle (eye icon, `lucide-react`) — 44×44px hit target per the spacing exception above. Default state: hidden (dots), matches platform convention. |
| Role selection (signup) | Use a segmented control or radio group (NOT a dropdown — only 2 options, radio/segmented is faster to scan and select). Selecting "Librarian" reveals the invite-code field inline (no page navigation) — progressive disclosure, not a modal. |
| Form validation | react-hook-form + zod, validate on blur AND on submit (not on every keystroke — keystroke-level validation feels naggy on password fields in particular). Show one error message per field, directly below the field, in Destructive red at Label size (14px). |
| Protected route / role-gated dashboard stub | Two minimal stub views differentiated by role (see Empty State Body copy above) — exists primarily to PROVE the `ProtectedRoute` + `require_role` server-side enforcement (success criterion #4), not to deliver real dashboard functionality. Keep visually minimal: page title (Display), welcome text (Body), and — if role is librarian — a visibly distinct element (e.g., a "Librarian tools" label/badge in the accent color) so a screenshot alone proves role differentiation during verification. |
